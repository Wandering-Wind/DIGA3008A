Final Reflection on the Implementation of the Website

FOLDER STRUCTURE

Does my folder structure reflect the hierarchical information structure of my content?
Yes, my folder structure mirrors the hierarchy of the site content. Each main section of the website (portfolio, designs, blogs) is organized into its own subfolder, which supports clear navigation and scalability. Assets such as my images are also grouped logically. All images go to one folder, where there are subfolders for each section of the website that requires images (e.g., Images folder > Portfolio > Game_Analysis > air_hockey.png). This helps maintain a meaningful structure that aligns with my site layout and user journey. The root of the project contains the index files, and each section branches off into folders from there.

Does it separate concerns: HTML, JavaScript, and CSS (data layer, logic layer, and presentation layer)?
Yes, the website follows separation where concerned. HTML files are placed in the root or sections folders and contain content and semantic markup. The main CSS file that is shared by every section is stored in the "Styles" folder, whereas particular styling that should be different to each page or section is in their respective folders (e.g. designs.css will be in the Designs folder with the designs.html) JavaScript files follow the same layout as CSS layouts. Main shared JS is located in the "Scripts" folder, whereas the rest are organized by their respective sections to handle logic and interactivity. This separation ensures better maintainability and clarity for me.

Is it modular and scalable?
Yes, the structure is modular and uses consistent naming conventions. File names have the first letter Uppercase and words seperated by a "_". File names are all lowercase, separated by "_" and follow the snake case naming convention. This convention increases readability. Components such as cards or navigation elements are designed to be reusable. This modularity makes the site easy to scale as more sections and features are added, and the naming conventions support both human readability and long-term maintenance. Images follow a convention according to where they belong: blog_week_7_dischem_con4 (pros and cons images).

SEMANTIC MARKUP

Is my markup providing sufficient context and information to the browser for my user?
Yes, mostly. Elements like <header>, <main>, <article>, <section>, <nav>, <footer>, <figure>, <figcaption>, and <time> are examples of semantic HTML being correctly used. The use of meta tags (author, description, viewport) also contributes to improved context and SEO.

Are my interface elements expressed fully and correctly?
Mostly, yes. Navigation elements are appropriately grouped using <nav>, with clearly labelled links. The layout for the burger menu is solid for responsiveness. Buttons are correctly named and marked <button> for readability (e.g. #toTopBtn).

Is my content expressed fully and correctly?
Yes, to the best of my ability. The use of <article> and <h1> with h-entry, p-author, dt-published (microformats) enhances semantic meaning and machine readability. Each section, like User Alignment, User Experience, and References, is logically structured with appropriate headings (<h3>) and paragraphs. All content is expressed fully, and no content is cut off by layout or anything like that.

What usability can be augmented by the use of semantic markup?
Keyboard navigation and screen reader usability are greatly enhanced. For example, <nav> helps assistive tech skip to menus, and <article> allows screen readers to announce blog content distinctly. Semantic HTML improves SEO, ensuring crawlers can better understand my structure and rank my content accordingly. <figure> and <figcaption> together help describe visuals in context, aiding both human and machine users.

Where can the use of semantic markup improve accessibility?

  • <header>, <main>, <footer>: Defines document regions for screen readers.
  • <time datetime>: Communicates exact publishing date in machine-readable format.
  • <article class="h-entry">: Microformats help with web syndication, search engine display, and accessibility.
  • <a class="p-author h-card">: Supports microdata for authorship metadata.

METADATA

Am I including sufficient metadata to guide the browser in interpreting my content?
Yes. My metadata includes:

  • <!DOCTYPE html> - Correctly declares HTML5 document type.
  • <html lang="en"> - Sets the document language to English.
  • <meta charset="UTF-8"> - Ensures correct character encoding (UTF-8).
  • <meta name="viewport" content="width=device-width, initial-scale=1.0"> - Enables responsive design.
  • <meta name="author" content="Amina Moosa"> - Provides authorship information.
  • <meta name="description" content="Blog week 3 about Interaction Design and a web-style guide"> - Supports SEO and sharing.
  • <title> is descriptive and accurate for that specific page content.

Have I identified content that requires metadata?
Yes, mostly. I've set the page language: <html lang="en">. I've set the author, myself, Author: <meta name="author" content="Amina Moosa">
Other content requiring metadata is appropriately marked up, for example:

  • Blog post metadata is expressed using Microformats (h-entry, p-name, p-author, dt-published, etc.) – This is great for semantic clarity and can be picked up by search engines and assistive tech.
  • <link rel="icon" href="..."> – Gives my site a favicon.
  • <link rel="stylesheet" href="..."> and <script src="..."> – External resources are correctly linked.
  • Font loading uses preconnect, improving performance.

Is my metadata expressed fully and correctly?
Yes, mostly. My metadata tags are placed within the <head> element. The attributes are spelled and closed properly. I make use of semantic microformats (e.g., h-entry, p-author, dt-published). My font loading and icons are optimized via preconnect. External links are marked as class="u-url" in references.

CSS

Am I avoiding in-line styles and using CSS to create consistent styling throughout my website?
Yes, mostly. Most styling uses CSS, such as the font and layout or colour. This promotes reusability and helps with cleaner code. I use classes and IDs defined in external CSS sheets.

Is it modular and retainable?
Yes, mostly. My CSS is modular since it improves scalability through the use of responsive design for pages and images. It is also maintainable, since all the repeated styling appears in the main CSS to avoid repetition and prevent styles from breaking when adding new features.

Am I using responsive layouts that adapt to different screen sizes and devices?
Yes, mostly. This matters since users visit websites on phones, tablets, and laptops, and my layout is flexible enough to adapt. I use media queries to adjust the layout for different viewports. I set widths in percentages where appropriate.

JAVASCRIPT

Am I expressing interaction points and functionality using JS? (through event handling and DOM manipulation?)
I’m effectively using event listeners and DOM manipulation to handle interaction points. DOMContentLoaded: Multiple instances to ensure the DOM is loaded before attaching behaviour such as transitions between pages. Button click: .readmore-btn and navigation buttons (prevBtn, nextBtn) trigger logic on click. SVG animation: I’m dynamically creating and animating SVG elements for the page beam effect, etc.

Event handling example:
There is one click event listener bound in my JS file: button.addEventListener("click", function () {   window.scrollTo({ top: 0, behavior: "smooth" }); });
This is inside the DOMContentLoaded event handler for the element with id "toTopBtn." It triggers smooth scrolling to the top of the page when that button is clicked.

Which interaction points and functionalities are repeated? Can they be separated out and consolidated?
Interaction points and functionalities that are repeated include the prevBtn and nextBtn, since they all require an array of files from that specific folder. The transitions that are repeated are slightly varied across sections. Other than that, interactions for specific sections are in their respective sections .js files.

Am I avoiding in-line scripting and linking through to the JS?
Yes. I'm avoiding inline scripting correctly, there's no inline JS in my HTML. All logic is externalized to .js files and executed using event listeners, which is good practice for maintainability, security (avoiding inline execution), and separation of concerns.

Am I using async requests to reduce redownloading and improve performance and UX? Yes, for my API, since it improves performance and reduces full-page reloads.
